### 2.2 分层架构设计

```
Client → HTTP Server → RPC → TCP Server → MySQL/Redis
```

**优势:**
- 职责清晰,易于维护
- HTTP Server无状态,可水平扩展
- TCP Server集中业务逻辑

## 4. 关键技术决策

### 4.1 为什么自研RPC而非gRPC?

**原因:**
1. 学习目的:深入理解RPC原理
2. 轻量级:无需proto编译
3. 灵活性:完全掌控协议设计
4. 简化部署:无额外依赖

**权衡:**
- ✅ 优点:简单、灵活、易调试
- ❌ 缺点:性能略低于gRPC、无双向流

### 4.2 为什么Session存Redis?

**原因:**
1. 高性能:内存存储,微秒级响应
2. 天然分布式:支持多实例共享
3. TTL支持:自动过期清理
4. 成熟稳定:久经考验

**权衡:**
- ✅ 优点:速度快、可扩展
- ❌ 缺点:需要额外维护Redis

### 4.3 为什么选择Gin框架?

**原因:**
1. 高性能:最快的Go Web框架之一
2. 生态完善:中间件丰富
3. API友好:易于上手
4. 社区活跃:问题易解决

## 5. 遇到的挑战与解决方案

### 5.1 挑战1: 1000万数据插入慢

**问题:** 单条插入耗时过长,预计需要数小时

**解决方案:**
- 批量插入:每批1000条
- 并发处理:10个goroutine并行
- 关闭日志:减少IO开销
- 调整MySQL参数:innodb_buffer_pool_size

**效果:** 插入速度从500条/秒提升到5000条/秒

### 5.2 挑战2: 高并发下连接池耗尽

**问题:** 2000并发时,RPC连接池频繁耗尽

**解决方案:**
- 扩大连接池:100 → 300
- 连接复用:使用defer归还连接
- 超时控制:避免连接长时间占用
- 监控告警:及时发现瓶颈

**效果:** 连接池利用率从95%降到70%

### 5.3 挑战3: 随机用户场景性能下降

**问题:** 随机用户导致Redis缓存命中率低

**解决方案:**
- 增加Redis内存:512MB → 2GB
- LRU策略:自动淘汰冷数据
- 预加载热点:启动时加载常用用户
- 本地缓存:二级缓存减少Redis访问(待实现)

**效果:** QPS提升15%,延迟降低20%
